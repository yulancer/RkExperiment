# R.K experiment on .NET 6

Готовое демо-решение для проверки поведения `Rebus.Kafka` в сценарии:

- 1 сервис-публикатор
- 1 сервис-подписчик типа **A**
- 2 экземпляра сервиса-подписчика типа **B**

Сценарий моделирует:

- отдельную МС **A**, которая должна получать **каждое** событие
- МС **B** в двух экземплярах, которые должны **делить нагрузку** между собой

## Состав решения

- `RkExperiment.Contracts` — общие контракты
- `RkExperiment.Publisher` — публикует `DemoEvent`
- `RkExperiment.ConsumerA` — подписчик A, отдельная consumer group
- `RkExperiment.ConsumerB` — подписчик B, один и тот же бинарник запускается в 2 экземплярах
- `RkExperiment.Admin` — вспомогательная утилита для явного создания топика с нужным количеством партиций
- `docker-compose.yml` — Kafka, Kafka UI и сервисы эксперимента

## Идея эксперимента

Используются **две разные consumer group**:

- `rk-consumer-a`
- `rk-consumer-b`

Оба сервиса подписываются на один и тот же тип события `DemoEvent`.

Ожидаемое поведение:

- `consumer-a` получает **все** сообщения
- `consumer-b-1` и `consumer-b-2` вместе получают **все** сообщения, но **каждое конкретное сообщение обрабатывает только один экземпляр группы B**

## Быстрый запуск

### 1. Поднять Kafka и подписчиков

```bash
docker compose up --build kafka kafka-ui consumer-a consumer-b-1 consumer-b-2
```

### 2. Отправить пачку событий

```bash
docker compose --profile publisher up --build publisher
```

### 3. Посмотреть логи

```bash
docker compose logs -f consumer-a consumer-b-1 consumer-b-2 publisher
```

### 4. Открыть UI Kafka

Открой `http://localhost:8080`

Там можно посмотреть:

- какие топики создались
- сколько у них партиций
- какие есть consumer groups
- как распределяются сообщения

## Что смотреть в логах

### Проверка pub/sub

У `consumer-a` должны появиться **все** `seq=1..N`.

### Проверка scale-out для B

У `consumer-b-1` и `consumer-b-2` суммарно должны быть **все** `seq=1..N`, но без дублей между экземплярами одной группы.

Пример ожидаемой картины:

- `consumer-a` обработал `1,2,3,4,5,6...`
- `consumer-b-1` обработал `1,3,5,7...`
- `consumer-b-2` обработал `2,4,6,8...`

Точное распределение зависит от ребалансировки и ключей Kafka.

## Варианты эксперимента

### Эксперимент 1. Автосоздание топиков

По умолчанию в примере включено:

- на брокере: `KAFKA_CFG_AUTO_CREATE_TOPICS_ENABLE=true`
- у клиентов: `Kafka__AllowAutoCreateTopics=true`

Для проверки:

1. удалить контейнеры и тома
2. поднять только Kafka и подписчиков
3. открыть Kafka UI и убедиться, что топики появляются автоматически после подписки/публикации

### Эксперимент 2. Отключить автосоздание

Поменять:

- в `docker-compose.yml` у Kafka: `KAFKA_CFG_AUTO_CREATE_TOPICS_ENABLE='false'`
- у потребителей: `Kafka__AllowAutoCreateTopics='false'`

После этого топики нужно создавать явно.

### Эксперимент 3. Явно создать топик с нужным числом партиций

Вариант через .NET-утилиту:

```bash
docker build -t rk-admin --build-arg PROJECT=src/RkExperiment.Admin/RkExperiment.Admin.csproj .
docker run --rm \
  -e KAFKA_BOOTSTRAP_SERVERS=host.docker.internal:9094 \
  -e TOPIC_NAME=DemoEvent \
  -e TOPIC_PARTITIONS=6 \
  rk-admin
```

После этого поднять подписчиков и паблишер.

### Эксперимент 4. Изменение количества экземпляров B на лету

1. поднять `consumer-b-1`
2. отправить сообщения
3. поднять `consumer-b-2`
4. снова отправить сообщения
5. посмотреть ребалансировку группы `rk-consumer-b`

Команда:

```bash
docker compose up --build kafka kafka-ui consumer-a consumer-b-1
# после этого отдельно
docker compose up --build consumer-b-2
```

## Что именно проверяет решение

### 1. Автоматизация создания топиков

Проверяется запуском с включенным `AllowAutoCreateTopics` и `auto.create.topics.enable`.

### 2. Автоматизация настройки partitions

Проверяется двумя способами:

- дефолтное число партиций брокера (`KAFKA_CFG_NUM_PARTITIONS`)
- явное создание через `RkExperiment.Admin` с `TopicSpecification.NumPartitions`

### 3. Изменение настроек топиков при добавлении/удалении подписчиков в реальном времени

Это решение показывает, что при добавлении/удалении экземпляров происходит **rebalance consumer group**.

Но само приложение **не меняет автоматически настройки топиков** при масштабировании. Для проверки нужно смотреть:

- меняется ли распределение партиций между экземплярами
- меняется ли число партиций самого топика

### 4. Подход к организации топиков

На практике этот эксперимент позволяет увидеть отдельно:

- input topic / queue конкретного сервиса
- event topic для публикуемого типа события `DemoEvent`

То есть можно оценить, ближе ли транспорт к модели:

- `1 topic = 1 service input`
- `1 topic = 1 event type`
- гибрид этих подходов

## Полезные команды

Остановить всё:

```bash
docker compose down
```

Остановить всё и удалить данные Kafka:

```bash
docker compose down -v
```

Повторно отправить события:

```bash
docker compose --profile publisher up publisher
```

## Замечания

1. Для сервиса A и группы B специально заданы **разные group id**.
   Это важно: только так A получает каждое сообщение независимо от B.

2. У двух экземпляров B задан **один и тот же group id**.
   Это важно: только так они делят сообщения между собой как одна масштабируемая МС.

3. В примере обработчики однопоточные (`SetNumberOfWorkers(1)`, `SetMaxParallelism(1)`), чтобы логи было проще анализировать.

4. Для реального исследования удобно менять:

- `Publisher__Count`
- `Publisher__DelayMs`
- `Service__ProcessingDelayMs`
- `TOPIC_PARTITIONS`

## Что стоит зафиксировать по итогам прогона

Рекомендую после запуска отдельно записать:

- какие топики реально создались
- сколько у них партиций
- кто их создал: брокер, transport или admin-утилита
- как ведёт себя группа `rk-consumer-b` при старте/остановке второго экземпляра
- простаивают ли экземпляры при числе партиций меньше числа consumers
- как называются event topic и input topic на практике
